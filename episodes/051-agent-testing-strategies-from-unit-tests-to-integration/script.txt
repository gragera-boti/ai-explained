Agent testing strategies — From unit tests to integration

Testing a regular app is straightforward. You write a unit test that says "when I call this function with X, I expect Y." You run it a hundred times, and if it passes a hundred times, you're confident it works. Now try that with an AI agent. The same prompt can produce different tool calls, different reasoning paths, and different final answers. Testing an agent isn't about checking exact outputs — it's about verifying behaviors, boundaries, and robustness under uncertainty.

Let's walk through the testing pyramid for AI agents, from the simplest unit tests all the way up to full integration testing.

At the bottom of the pyramid is tool testing. Every tool your agent can call — whether it's a search function, a database query, a calculator, or an API call — should have its own unit tests. Does the tool handle edge cases? Does it return the right schema? Does it fail gracefully when the API is down? These tests are deterministic and easy to write. You're testing the tool, not the agent. This is your safety net. If the tools are broken, nothing above them will work.

Next up is prompt testing. This is where things get interesting. You want to verify that the agent's system prompt produces the right behavior for different scenarios. Test that the agent follows instructions about when to call a tool versus when to answer directly. Test that it doesn't call tools it shouldn't have access to. Test that it formats its thoughts correctly. Since LLMs are non-deterministic, you don't check exact output — you use evaluation metrics. Does the response contain the required tool call format? Does it avoid prohibited patterns? You run each test multiple times and look for consistency.

The pattern here is property-based testing adapted for LLMs. Instead of testing for a specific output, you test for properties of the output. "The response should contain exactly one tool call." "The tool name should be one of the approved list." "The agent should not reveal its system prompt." Each property is checked with an LLM-as-judge evaluation or a simple regex pattern. You run the test multiple times — say ten to twenty iterations — and check that the property holds above some threshold, like ninety-five percent of the time.

Above prompt tests comes conversation flow testing. This is where you simulate a full interaction and check that the agent handles a sequence of turns correctly. For example: user asks a question, agent calls a tool, tool returns data, agent asks a clarifying question, user provides clarification, agent calls another tool, and finally delivers an answer. You're testing that the agent maintains context across turns, that it remembers what tools it already called, and that it doesn't loop indefinitely.

Conversation flow tests are the trickiest because they require simulating both the user and the tool responses. The standard approach is to write a test harness that plays the role of the user according to a script and mocks the tool responses. This lets you test specific scenarios deterministically — the tool always returns the data you specify — so you can focus on whether the agent navigates the conversation correctly.

Now for the hardest layer: behavioral testing. This is where you test the agent against open-ended scenarios to see if it does the right thing when there's no single correct answer. For example, you give the agent a vague request like "help me plan a team offsite" and check that it asks about budget, headcount, dates, and preferences before making recommendations. There's no single right response, but there are behaviors you expect — asking the right questions, considering constraints, offering reasonable options.

Behavioral testing is where you really need LLM-as-judge evaluations. You define rubrics: "Does the agent ask about budget? Does it consider dietary restrictions? Does it propose multiple options?" Then you have a judge model score each run against these rubrics. You run dozens or hundreds of test cases and aggregate the scores. This gives you a statistical picture of whether your agent is getting better or worse as you change prompts, tools, or the model itself.

Then there's adversarial testing. This is where you deliberately try to break your agent. Prompt injection attacks — can a user trick the agent into ignoring its instructions? Tool misuse — can a user make the agent call a tool with malicious parameters? Resource exhaustion — can a user cause the agent to loop or run up costs? Adversarial testing should be an ongoing practice, not a one-time check. Every time you add a new tool or change the system prompt, you should try to break it.

Finally, at the top of the pyramid, you have integration testing. This is testing the agent against the real world — actual APIs, real databases, production-adjacent environments. Integration tests catch the things mocks miss: rate limits, network errors, slow responses, unexpected data formats. They're slower and more expensive, so you run fewer of them, but they're essential for catching the real-world failures that mock tests never will.

A practical testing strategy for most agent projects looks like this. Have a fast suite of tool and prompt tests that run in seconds — these catch regressions immediately. Have a medium suite of conversation flow tests that run in a few minutes — these catch logical errors in your agent's behavior. Have a small suite of behavioral and integration tests that run on a schedule or before releases — these catch the subtle, real-world issues.

The most important thing to remember is that agent testing is never done. Agents are generative systems. Every new model release, every prompt change, every new tool can introduce unexpected behaviors. Build your testing pipeline early, make it easy to add new test cases, and treat regressions as learning opportunities about how your agent behaves under different conditions.
