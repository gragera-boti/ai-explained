Agent Communication Protocols — Structured Message Passing Between Agents

Imagine you're at a busy office. The marketing manager needs the sales team's latest numbers, the developer needs design specs from the product team, and the CEO wants a summary from every department. If everyone just shouted into the open, nothing would get done. Instead, they use structured communication — memos, emails, Slack messages with clear subjects, headers, and recipients. That's exactly why AI agents need communication protocols. When multiple agents work together, they can't just pass random chunks of text back and forth. They need a structured language for asking, answering, delegating, and reporting.

Communication between agents might sound simple — just have one agent output text and another read it. But that breaks down fast. Without structure, an agent doesn't know whether a message is a question, an instruction, a result, or an error. It can't tell which task a result belongs to. It has no way to request clarification. And if the conversation gets long, the agent gets lost in a sea of unstructured text. Communication protocols solve all of this by defining a shared format that every agent understands.

The most basic protocol is request-response. One agent sends a structured request with a clear intent — "I need you to research this topic and return a summary." The other agent processes it and sends back a structured response with a status — "Completed" or "Failed" — plus the actual result. This is simple, reliable, and works great for one-off task delegation. Frameworks like LangGraph and CrewAI use this under the hood when agents hand off work to each other.

But real-world agent workflows get more complex. An agent might need to report progress on a long-running task, ask for missing information, or escalate a problem it can't solve. That's where richer protocols come in. Some protocols support streaming intermediate results — the researcher agent sends back found documents as it goes, rather than waiting until it's done. Others support negotiation — two agents debate the best approach before either starts working. And some support error handling with retry semantics, so if an agent returns "I don't have access to that database," the caller can route the task to a different agent.

Message format matters too. A structured message usually includes a sender ID, a recipient or routing hint, a message type (request, response, error, progress update, clarification, etc.), a correlation ID that ties the message to a specific task or conversation thread, a payload with the actual content, and metadata like timestamps and priority. This is very similar to how HTTP requests have headers, methods, and bodies — the structure lets intermediaries route messages, retry failures, and log everything without touching the content.

The A2A protocol from Google is a real-world example of this thinking. It defines a standard for agent-to-agent communication with structured messages, capabilities discovery (agents advertise what they can do), and task management with a shared task state machine. An agent sends a request, and the receiving agent can accept, reject, or delegate it. Both agents can check the task's status at any time. This lets agents from different systems and different providers interoperate — like how HTTP lets any web browser talk to any web server.

MCP, the Model Context Protocol, is another example, though it focuses on agent-to-tool communication rather than agent-to-agent. Still, the idea is the same: structured messages prevent misunderstanding. When an agent calls a tool via MCP, the tool knows exactly what format the input should be in, what parameters are required, and what a successful response looks like.

Why does this matter for you as someone building with AI agents? Because the complexity of your agent system determines the protocol you need. If you have two agents working sequentially on a simple pipeline, a basic request-response message is fine. But if you have a team of a dozen agents with parallel workflows, error recovery, and dynamic task routing — you need something closer to A2A. The more moving parts, the more structure you need.

The golden rule of agent communication is simple: don't let your agents guess what a message means. Define your protocol upfront. Specify message types, required fields, and expected responses. Add correlation IDs so you can trace which task each message belongs to. And always include a way for agents to say "I don't understand" — a clarification request. That last one is the most overlooked and the most important. When an agent is confused, you want it to ask for help, not silently do the wrong thing.
