Agents and Tool Use — AI That Can Take Actions

For most of their history, language models have had one superpower and one glaring limitation. The superpower? They can generate remarkably fluent, intelligent-sounding text. The limitation? That's all they can do. Ask a language model what the weather is, and it'll give you a confident-sounding guess based on its training data — not an actual forecast. Ask it to book a flight, and it'll write a lovely paragraph about how to book a flight. Helpful, but not the same thing.

AI agents change that equation entirely. An agent is a language model that can actually do things in the world — not just talk about doing things.

So what exactly makes a model an "agent"? At its core, it's three ingredients. First, the model can perceive its environment — it receives information from external sources. Second, it can take actions — it calls tools, runs code, sends messages. And third, it operates in a loop — it observes the results of its actions and decides what to do next. That loop is what separates an agent from a simple chatbot. A chatbot gives you one response and waits. An agent keeps going until the job is done.

Let's make this concrete with an analogy. Imagine you're at a restaurant. A regular language model is like a very knowledgeable food critic sitting next to you. They can describe every dish on the menu in exquisite detail, suggest wine pairings, and tell you the history of French cuisine. But they can't actually order for you, call the waiter, or pay the bill. An agent, on the other hand, is like having a personal assistant at the table. They read the menu, ask you what you're in the mood for, flag the waiter, place the order, and handle the check. Same knowledge, but now connected to actions.

The technical mechanism that makes this work is called tool use, sometimes called function calling. Here's how it works. When you set up an agent, you give the language model a list of tools it can use — think of them as a toolkit. Each tool has a name, a description, and a schema that defines what inputs it expects. For example, you might define a tool called "get_weather" that takes a city name and returns the current temperature. Or a tool called "search_database" that takes a query and returns matching records.

When you send the model a message like "What's the weather in Madrid?", the model doesn't just generate text. Instead, it recognizes that it has a relevant tool available. It outputs a structured request — essentially saying, "I'd like to call get_weather with the input Madrid." Your application catches that request, actually runs the function — hitting a real weather API — and feeds the result back to the model. The model then incorporates the real data into its response: "It's currently 18 degrees and sunny in Madrid."

This back-and-forth — model requests a tool call, application executes it, result goes back to the model — can happen multiple times in a single conversation. The model might need to call three different tools to answer a complex question, chaining them together. That's where things get powerful.

Now, there's an important architectural distinction that Anthropic highlights in their work on building agents. Not every system that uses tools is truly an "agent." There are workflows, and there are agents. A workflow is a system where the steps are predefined by the developer — the model fills in the blanks, but the overall structure is fixed. Think of it as a flowchart where the boxes are already drawn, and the model just decides what goes in each box. An agent, by contrast, is a system where the model itself decides what steps to take, which tools to call, and in what order. The model is driving. It plans, executes, observes, and adapts.

Both patterns are useful, and in practice, many real systems blend the two. You might have a workflow that handles the high-level structure — first classify the customer's request, then route it — but within each step, an agent decides how to gather information and respond.

Here's a real-world example of tool use in action. Consider a customer support agent for an e-commerce company. The model has access to tools like "lookup_order," "check_inventory," "process_refund," and "send_email." A customer writes in saying their package never arrived. The agent calls lookup_order with the customer's ID, sees the order was marked as delivered, checks the shipping carrier's tracking data using another tool, discovers the package was left at a wrong address, initiates a replacement through process_refund, and sends the customer a confirmation email — all in one interaction. No human needed to orchestrate those steps. The model figured out the sequence on its own.

Tool use isn't limited to simple API calls, either. Some of the most powerful agent applications involve code execution — the model writes and runs code to solve problems. Others involve browsing the web, reading files, querying databases, or even controlling other software. The model's "tools" can be anything your application exposes.

There are real challenges here, though. Agents can make mistakes, and unlike a chatbot that just gives a wrong answer, an agent that makes a mistake might actually do something — send the wrong email, delete the wrong file, charge the wrong amount. That's why robust agent design includes guardrails: confirmation steps for high-stakes actions, validation checks on tool inputs, and human-in-the-loop approval for anything irreversible.

Another challenge is reliability. The more steps an agent chains together, the more opportunities for things to go wrong. If each step has a 95 percent success rate and you chain ten steps, your overall success rate drops to about 60 percent. This is why the best agent builders keep things as simple as possible — using the minimum number of steps and the simplest architecture that gets the job done.

The landscape of agent frameworks has exploded over the past year. There are SDKs from Anthropic, OpenAI, AWS, and many others. But a consistent finding from teams building production agents is that simpler is better. Many of the most successful agent deployments use direct API calls with a handful of well-defined tools, rather than complex orchestration frameworks.

What's exciting about agents and tool use is that they represent a fundamental shift in what AI can be. Language models started as text generators. With tool use, they become general-purpose reasoning engines that can interact with the world. They can check facts against real data, take actions on your behalf, and adapt their approach based on what they learn along the way.

We're still early. Today's agents are impressive but brittle. They work best in well-defined domains with clear tools and reasonable guardrails. But the trajectory is clear — AI is moving from systems that talk about doing things to systems that actually do them. And that changes everything.
