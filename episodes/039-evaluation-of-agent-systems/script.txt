Evaluation of Agent Systems — Measuring What Matters

If you've ever tried to judge whether an AI agent is actually good at its job, you've run head-first into a difficult question: how do you measure something that has to make dozens of decisions, use tools, recover from mistakes, and adapt to unexpected situations? Evaluating a language model is simple — you give it a test, check the answers. Evaluating an agent is a whole different beast.

Let's start with why agent evaluation is harder. A regular LLM benchmark gives the model a prompt and checks the response. Pass or fail. Done. But an agent doesn't just answer — it acts. It calls tools, follows multi-step plans, interacts with environments, and sometimes has to recover when things go wrong. A single agent run might involve ten tool calls, five decision points, and a branching path where one wrong choice early on derails everything. How do you score that?

The most direct approach is end-to-end task completion. You give the agent a goal — "book a flight from New York to London on July 15th" — and you check whether it succeeded. Did the right flight get booked? Was it on the right date? This is called success rate, and it's the simplest evaluation metric. The agent either completes the task or it doesn't. But this has a big blind spot: an agent could succeed through luck, or fail through no fault of its own if the environment changes.

That's where progressive evaluation comes in. Instead of just checking the final outcome, you evaluate each step along the way. Did the agent choose the right tool? Did it pass the correct parameters? Did it recover gracefully when an API returned an error? This gives you much richer information. You might find that an agent has a 90% success rate on final tasks but consistently picks suboptimal tools — a pattern you'd miss entirely with end-to-end evaluation alone.

Several benchmark suites have emerged specifically for agents. AgentBench evaluates agents across diverse environments including operating systems, databases, web browsing, and games. SWE-bench tests agents on real GitHub issues — can the agent find the bug, write a fix, and pass the tests? WebArena drops agents into simulated web environments and asks them to complete tasks like "find the cheapest flights and book one." GAIA tests general AI assistants with questions that require multi-step reasoning and tool use. Each of these benchmarks captures a different dimension of agent capability.

But benchmarks have their own problems. An agent might score well on AgentBench but fail in the real world because real APIs behave differently than simulated ones. There's also the issue of reproducibility — agents that use language models are non-deterministic, so the same agent might succeed five times and fail five times on the same task. This means you need many runs to get statistically meaningful results, which is expensive and slow.

The research community has been developing more sophisticated evaluation approaches. One promising direction is trajactory-level evaluation using trained evaluator models — essentially using another LLM to judge whether each step in the agent's reasoning and action chain was correct. This is similar to LLM-as-judge for regular evaluations, but adapted for the sequential nature of agent behavior. Another approach is adversarial evaluation, where you deliberately introduce failure modes — flaky APIs, misleading instructions, missing information — and see how well the agent handles them.

Safety evaluation adds another layer entirely. You need to check that the agent doesn't take destructive actions even when given ambiguous instructions. Does it call dangerous tools without confirmation? Does it follow instructions to delete files or transfer money? These are evaluated through red-teaming: deliberately crafting prompts that try to trick the agent into doing something harmful.

There's also the question of efficiency. A perfect agent that takes twenty minutes and costs fifty dollars in API calls might be less useful than a good-enough agent that finishes in two minutes. Metrics like cost per task, latency, and number of tool calls per completion are increasingly important evaluation dimensions.

So what should you actually evaluate? If you're building an agent system, here's a practical framework. First, evaluate task success — did the agent do what was asked? Second, evaluate process quality — did it use the right tools in the right order? Third, evaluate robustness — how does it handle errors, unexpected inputs, and edge cases? Fourth, evaluate safety — does it stay within its guardrails? Fifth, evaluate efficiency — is it fast and cost-effective enough for production?

The key insight is that agent evaluation is a multi-dimensional problem. You can't reduce it to a single number. The best evaluation systems combine automated benchmarks with human review, especially in high-stakes domains. And as agents become more capable, the evaluation challenge only grows — the more autonomy you give an agent, the more important it is to measure not just what it does, but how it does it.

That's evaluation of agent systems. Test rigorously, measure honestly. See you next time.
